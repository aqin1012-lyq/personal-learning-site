## 这篇想解决什么问题

很多人刷二叉树递归题时，都会反复卡在一个很像细节、其实是核心的问题上：

- 为什么有的递归结果可以直接 `return dfs(...)`
- 有的题却必须先用变量接住左右子树结果，再在当前层做判断

如果这个点没想清楚，代码就会写得很凭感觉。
而一旦想清楚，你会发现很多树题其实都能被归到两类。

## 先说结论：递归大致有两种工作模式

### 第一种：传话型递归

特点是：
当前节点自己不怎么加工信息，而是根据题目结构，把答案从某个子树一路传上来。

这种情况下，通常可以直接：

```java
return dfs(root.left);
```

或者：

```java
return dfs(root.right);
```

因为子问题的返回值，本身就已经是你要的答案。

### 第二种：加工型递归

特点是：
当前节点必须同时看左右子树返回什么，再决定自己要返回什么。

这种情况就不能只写一个直接 return 了，而是通常要这么写：

```java
TreeNode left = dfs(root.left);
TreeNode right = dfs(root.right);
```

然后在当前节点做综合判断。

## 用最近公共祖先两题来感受差别

最典型的对比，就是：

- BST 最近公共祖先
- 普通二叉树最近公共祖先

它们表面题意很像，但递归结构完全不是一回事。

## 场景一：BST 最近公共祖先为什么可以直接 return

在二叉搜索树里，节点有大小关系：

- 如果 `p` 和 `q` 都比当前节点小，就去左边
- 如果都比当前节点大，就去右边
- 否则当前节点就是分界点

所以逻辑是：
你并不需要“左右都搜一遍再综合”，因为树本身已经告诉你应该往哪边走。

也就是说，当前层更像一个路由器：

- 判断方向
- 把问题交给对应子树
- 直接把子树答案传回来

这就是典型的传话型递归。

## 场景二：普通二叉树最近公共祖先为什么必须先接结果

普通二叉树没有 BST 的大小性质。
你不能提前知道答案在左边还是右边，所以必须：

- 左边找一次
- 右边找一次
- 再看当前节点和左右结果之间的关系

比如：

- 如果左边找到了，右边也找到了，那当前节点就是 LCA
- 如果只有一边找到了，那就把那一边返回上去
- 如果两边都没找到，返回空

这里的关键不是“递归调用了两次”，而是：
当前节点必须解释这两次调用的意义。

所以这就是典型的加工型递归。

## 真正该问自己的判断题

以后写递归时，可以先问自己一句：

**当前节点是不是必须利用左右子树的结果，额外做一次判断或组合？**

如果答案是：

- **不需要**，子结果就是最终答案的一部分，那大概率是传话型递归
- **需要**，必须在当前层加工、汇总、筛选，那大概率是加工型递归

这句判断，比背模板更有用。

## 一个更通用的理解方式

你可以把递归函数想成在向父节点汇报。

### 传话型

子节点说：
“我这里已经有答案了，你直接转给上面就行。”

### 加工型

子节点说：
“我这里只能给你局部信息，你得把左右两边放一起看，才能得出结论。”

这两种汇报方式，对应的代码结构自然就不一样。

## 为什么很多人会写混

因为题目表面都长得像：

- 先递归左边
- 再递归右边
- 再 return 某个东西

但真正决定写法的，不是“长相”，而是函数返回值的语义。

你得先想清楚：
这个递归函数返回的是——

- 一个完整答案
- 还是一个给父节点继续加工的中间信息

这个问题一旦想明白，很多二叉树题的递归骨架都会变稳定。

## 刷题时特别实用的一个习惯

开始写递归前，先把这句话补完整：

**`dfs(node)` 的返回值表示什么？**

比如：

- 表示“以 node 为根的子树中，目标节点的最近公共祖先”
- 表示“这棵子树里是否找到了目标”
- 表示“这棵子树的高度”
- 表示“这棵子树是否平衡，同时顺带返回高度”

一旦返回值定义清楚，你就更容易判断当前层是传话，还是加工。

## 最后一段总结

递归里“能不能直接 return”，本质上不是代码风格问题，而是信息流问题。

- 如果子调用返回的就是答案，当前层只负责转发，那就可以直接 return
- 如果当前层还要解释、组合、筛选左右结果，那就必须先接住，再处理

把这层想通之后，你写二叉树递归时会少很多“凭手感”，多很多结构感。
