## 这道题为什么常让人“转移会写，结果不对”

像“两个字符串的删除操作”这类题，很多人看完思路后会觉得自己已经懂了：

- 定义一个二维 `dp`
- 看两个字符是否相等
- 相等就继承，不等就取最优

结果代码一跑，答案却不对。

这种题里，真正最容易出问题的常常不是状态转移，而是初始化。

## 为什么初始化这么关键

因为二维字符串 DP 的第一行、第一列，不是随手填的边角料。
它们是在定义：

**当其中一个字符串长度为 0 时，问题答案是什么。**

而这件事，正好决定了整个表格后续推导的起点。

如果起点语义没立住，后面的转移即使形式上看起来没错，整张表也会被带偏。

## 这题本质在算什么

“两个字符串的删除操作”问的是：
把两个字符串删到完全相同，最少要删多少次。

你可以从两个角度理解它：

- 直接定义删除次数 DP
- 或者先求最长公共子序列，再推导删除次数

无论你走哪条路，初始化都绕不过去。

## 为什么第一行第一列这么容易写错

因为很多人写 DP 时，脑中其实没有一句清楚的话来描述 `dp[i][j]`。

比如你必须先说清楚：

- `dp[i][j]` 表示 word1 前 i 个字符和 word2 前 j 个字符变成相同所需的最少删除次数

一旦这个定义明确，初始化才有依据。

例如：

- `dp[0][j]` 表示第一个字符串为空，第二个字符串前 j 个字符要删多少次才能相同
- 那显然就是删掉这 j 个字符

同理：

- `dp[i][0] = i`

这不是“套路规定”，而是状态定义的自然结果。

## 字符串 DP 最该养成的习惯

每次做题前，别急着写循环，先回答三个问题：

1. `dp[i][j]` 表示什么
2. 当 `i = 0` 或 `j = 0` 时，这个定义意味着什么
3. 当前状态是从哪些更小状态推出来的

这三件事如果能提前说清楚，初始化基本就不会写歪。

## 为什么很多人会觉得“明明两种写法都像对的”

因为二维 DP 很容易出现两套下标体系：

- 一种是 `dp[i][j]` 对应前 `i` / `j` 个字符
- 一种是直接对应字符下标位置

这两种都能做，但初始化逻辑会跟着一起变。

问题往往不是哪种写法不行，而是：
你在第一种定义上写了第二种初始化，或者反过来。

所以真正要统一的不是代码形式，而是语义系统。

## 这题带来的更大启发

很多字符串 DP 题都共享同一个难点：

- 编辑距离
- 删除操作
- 最长公共子序列
- 最长重复子数组 / 子序列相关题

它们都在提醒你：

**DP 表格不是算出来的，它是先被“定义”出来的。**

初始化就是定义的第一块地基。

## 最后一段总结

“两个字符串的删除操作”这题最值得记住的，不是某个固定模板，而是一个判断标准：

如果你不知道第一行、第一列为什么这么填，那说明你对 `dp[i][j]` 的定义还没有真正想清楚。

字符串 DP 很多时候不是输在不会转移，而是输在状态和初始化没有对齐。
一旦这层对齐了，题目会顺很多。
